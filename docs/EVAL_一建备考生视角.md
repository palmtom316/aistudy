# 一级建造师备考生视角：aistudy 系统评价与改进意见

> **角色**：在职备考一级建造师（机电实务为主，兼公共科），每日有效学习约 1.5–2.5 小时。  
> **评价标准**：按「能不能靠它备考、能不能涨分」打分，不按工程师审美打分。  
> **对照基线**：仓库现状（约 2026-07）、SPEC/PLAN、样例 note/quiz、skills 与 dashboard。  
> **关联文档**：技术侧见 [`CODE_REVIEW.md`](CODE_REVIEW.md)；规格见 [`SPEC.md`](SPEC.md)；施工图见 [`PLAN.md`](PLAN.md)。

---

## 1. 结论（考生一句话）

**方向对、骨架硬，但离「能扛住一年一建备考」还差半截。**

本系统不是又一个刷题 App，而是「Obsidian 知识库 + Anki 间隔重复 + LLM 出题/复盘」的闭环。对机电实务这种「规范数值 + 工序 + 案例综合」特别对口；但当前内容几乎是样例级（1 篇 note、1 道 quiz、mock 教材），**工程完备度高、备考可用度低**。

| 状态 | 综合分（满分 10） | 含义 |
|------|-------------------|------|
| 当前 | **5.0** | 优秀骨架 × 空仓库 |
| 若完成「内容 MVP + 今日一键 + 题型对齐」 | **8.0+** | 可成小众但极强的一建实务系统 |

## 2. 作为考生最认可的 5 点

### 2.1 数据模型像「会考试的人」设计的

`core / mastery / exam_freq / last_reviewed / source` 直接对应备考真实问题：

| 字段 | 考生心声 |
|------|----------|
| `core` | 大纲核心还是边角料？ |
| `mastery` 0–3 | 没看过 / 看过 / 会做题 / 能讲清楚 |
| `exam_freq` | 真题出现过几次？ |
| `source` | 出处在哪一页、哪条规范？ |
| `effective_date` / `superseded_by` | 规范改了没？旧条款还在不在？ |

机电实务最怕「背了废规范」。dashboard §F 法规时效预警，比多数市面题库更懂建造师。

### 2.2 描述子词典贴实务记忆结构

受控 8 类：`定义 / 组成 / 分类 / 作用 / 工序 / 参数 / 要求 / 规范条文`。

一建实务记忆点本来就是「数值 + 条文号 + 工序链」。样例「电缆敷设」几乎是标准考点形态：

- **参数**：一般 ≥0.7m，农田 ≥1m  
- **规范**：GB 50168-2018 §5.1.3  
- **工序**：放线→开挖→铺砂→下缆→盖板→回填→标识  

比「整章复制粘贴笔记」更适合 Anki 成卡，也更适合案例踩分点拆解。

### 2.3 案例题被当成一等公民

一建实务**案例题往往决定过不过线**。系统把 `quiz`（单点）与 `cases`（多步、可量化 rubric、多知识点回链）拆开，且要求：

- ≥2 步、≥2 个知识点  
- 步骤给分、关键点扣分  
- 必须有 `source` 才能入库  
- 答错降 mastery，答对不自动升（防「蒙对就升难度」）

比「对了打勾、错了看解析」更贴近阅卷逻辑。

### 2.4 诊断 → 今日计划 → 错题复练闭环完整

链路意图清晰：

```
dashboard（诊断）
  → study-drill（今日 ≤6 项）
  → quiz / case（做题）
  → journal
  → study-review（周/月）
  → 再 drill
```

在职备考最缺的不是资料，是**今天该干什么、明天补什么、错题别丢**。

### 2.5 信任链对考试场景是正确姿态

LLM 出题必须回链教材/真题/「LLM-generated; reviewed 日期」；`correct:true` 不自动升 mastery。  
考生宁愿系统「保守、可追溯」，也不要「看起来懂其实是幻觉」。

## 3. 真实痛点（按严重程度）

### P0｜内容几乎空，没法开考

| 目录 | 现状（评审时） |
|------|----------------|
| `notes/` | 仅「电缆敷设」1 篇 |
| `quiz/` | 1 道 |
| `cases/` | 空 |
| `materials/` | 约 8 行 mock 教材 |
| `journal/` | 空 |

设计再漂亮，**没有覆盖率就没有分数**。当前只能当「方法原型」，不能当「备考主阵地」。

### P0｜冷启动太重，在职考生扛不住

完整链路依赖：Obsidian + Dataview +（PDF++）+ Anki + CLI/Make + LLM skills + 人审。

- 装插件、OCR、taxonomy、export Anki、SQL 导 CSV 再 sync……摩擦极高  
- 第一周容易死在「环境与流程」，而不是死在考点  
- 缺少「今晚打开就能练 20 分钟」的最短路径  

**冷启动成本 ≈ 中途放弃率。**

### P1｜一建考试形态覆盖不全

| 考试形态 | 系统现状 | 考生需求 |
|----------|----------|----------|
| 实务单选/多选 | quiz 偏开放问答 | 大量标准选择题 + 选项陷阱解析 |
| 实务案例 | case 设计到位，无真题卷结构 | 按「第 N 问」真题格式 + 历年卷 |
| 公共科（法规/管理/经济） | taxonomy 仅机电实务 | 三公科题量大，需整科扩展 |
| 计算题 | 弱 | 公式卡 + 分步演算 + 常见陷阱 |
| 识图/系统图 | 有 study-tikz，门槛高 | 常用系统图模板、标注考点 |

只练「写答案」不练「选对选项」，客观题会脱节。

### P1｜`exam_freq` 是灵魂字段，却几乎没数据

`study-drill` 已写明：全局 `exam_freq>0` 占比 &lt;20% 时**忽略高频排序**。  
没有「真题年份-题号-考点」整理落地，**优先级退化成 core+mastery**，丢掉「按真题权重刷」的顶级策略。

### P1｜Anki 同步路径太「工程师」

路径：Anki 复习 → SQL 导出 CSV → `anki-sync.sh` →（可选）study-sync 看 drift。  
考生预期是「刷完 Anki，掌握度自动更新」。多两步则同步率崩，dashboard 会撒谎。

### P2｜note 级 mastery vs 多张 Anki 卡语义模糊

一篇 note 多个 descriptor = 多张卡，mastery 却是 note 级标量。  
现实：可能「埋深参数」很熟、「整条工序」仍糊——系统只能给一个 mastery，「会不会做题」判断会偏。

### P2｜dashboard 绑 Obsidian；缺备考时间线

- 日常「一眼进度」依赖 Dataview；地铁轻量查看路径不够  
- 系统偏「长期学习」，一建是强 deadline：缺阶段目标（基础/强化/冲刺）、公共科 vs 实务配比、战役级计划  
- `study-drill` 解决「今天 6 项」，未解决「离考 90 天怎么排」

### P3｜样例质量暴露内容风险

教材为 mock、页码示意。信任链设计正确，但批量 LLM 生成若人审跟不上，会出现「有 source 字段、source 不可信」。  
考试场景里，**错一个规范数值比没背更致命**。

## 4. 优化改进意见

### 4.A 立刻提升备考可用度（建议 2–4 周）

#### A1. 机电实务「最小可考包」（内容 MVP）

不要先扩 CPA/医学。先做成：

- 按官方大纲：每章 ≥15–30 个原子 note  
- 每 note：3–6 个 descriptor + 1 道自测 + **真实** source 页码  
- 每章：≥10 道标准选择题 quiz + ≥2 道案例 case  
- 近 5 年真题：至少映射 `exam_freq` 与题号  

**考生验收**：连续 14 天，每天 drill 出 6 项且不是同一道「电缆埋深」。

#### A2. 增加「一建客观题」题型模板

扩展 `templates/quiz.md`（或并列模板）：

- 单选 / 多选 / 判断  
- 选项 A–D、正确答案、干扰项陷阱说明  
- 解析强制绑 `规范条文` 或 `参数` descriptor  
- Anki 导出支持选择题卡型（不只正反面描述子）

#### A3. 真题导入 SOP（激活 exam_freq）

固定流程建议：

```text
真题 PDF/文本
  → 标注年份题号
  → link 到 note
  → exam_freq++
  → source: 2023一建机电-案例二-第3问
```

没有这条，dashboard「高频未掌握」永远是空壳。

#### A4. 一条命令的「今日学习」

目标体验：

```bash
make today
# 输出：今日 6 项 + 对应 note/quiz 路径 + 预计时长
# 做完提示：journal / 是否 anki-sync
```

`outline` / `extract` / `taxonomy` 留给周末「建库日」；**工作日只暴露 drill + quiz + case**。

#### A5. Anki 同步一键化

`make anki-sync` 内部串：定位 Anki 库 → SQL → sync → 打印 mastery 变化摘要。  
考生不应需要记住 `.sql` 文件名。

---

### 4.B 贴合阅卷与得分结构（建议 4–8 周）

#### B6. 案例题按「真题卷面」结构化

建议 `cases` 正文固定骨架：

```markdown
## 背景（工程概况）
## 问题1（X分）
## 问题2（X分）
## 问题3（X分）
## 标准答案 / 评分 rubric（按问、按采分点）
## 我的答案
## 错因 / 失分点 / 触发复习的 notes
```

并支持：采分点关键词、半对规则、分问回链多 note。

#### B7. 工序类强制 procedure-flow

`type: case-pattern` 且含 `工序::` 时，强制附 `templates/structural/procedure-flow`，并生成「打乱顺序改错」题。

#### B8. 规范版本管理考生可见化

在既有 superseded 之上增加：

- 本年考试可能用的规范版本清单  
- 新旧对照 note（旧参数 → 新参数，标红）  
- 优先推 `effective_date` 落在考纲年份内的条款  

#### B9. mastery 与多卡聚合规则写清

最低成本方案：

- 该 note 下任一张卡连续低 ease → note 标 drift  
- 全部 descriptor 卡 interval 达标 + 近期 quiz 正确 → 才建议 `mastery ≥ 2`  

避免「背过定义却案例全军覆没」却显示已掌握。

---

### 4.C 降低门槛、提高坚持率

#### C10. README 首页写清三档使用模式

| 模式 | 谁用 | 当天只做什么 |
|------|------|----------------|
| 刷题模式 | 工作日晚上 | `make today` + Anki ~20 min |
| 建库模式 | 周末半天 | prep → outline/extract → 人审 |
| 冲刺模式 | 考前 30 天 | 模拟卷 + 错题 + 案例专项 |

避免「全功能并列」让人以为每天都要跑完全链路。

#### C11. 手机轻阅读路径

Obsidian Sync，或导出「今日包」（今日 note 正文 + 待做 quiz）。通勤以读为主，不要求跑 Make。

#### C12. 周报用人话

`study-review` 呈现层建议固定：

- 本周正确率、案例均分  
- 失分最多的 3 个考点（带规范号）  
- 下周只攻这 3 个 + 2 道旧错题  
- 距建议模考还有 N 天  

护栏「只读结构化字段」可保留；**呈现要翻译成备考决策**。

#### C13. 谨慎扩展建造师 Descriptors 白名单

可评估增加（仍白名单，禁止 LLM 自造）：

- `验收`  
- `安全`  
- `质量通病`  
- `计算`  

实务高频维度不止当前 8 个，但必须守白名单纪律。

---

### 4.D 中长期

#### D14. 按一建报名结构扩科

1. 机电实务（或目标专业实务）内容做满  
2. 法规 → 管理 → 经济（题型不同，模板要分）  
3. 再考虑二建 / 其它 domain  

taxonomy 锁死 MVP 科目正确；需明确「解锁公共科」里程碑。

#### D15. 模考与粗粒度得分反馈

增强 `study-drill` 模拟卷模式：

- 真题建议用时  
- 客观题正确率 / 案例采分率分列  
- 薄弱章节雷达（对齐大纲章节）  
- 可选：core 覆盖 × 正确率 → 过线粗估（不装神棍）

#### D16. 校验前置为「人审助手」

在 `validate-content` / `taxonomy-check` 上补充：

- 数值类 descriptor 无 source → 拒收  
- case 缺分步 rubric → 拒收  
- note 无「自测一题」→ 警告  
- `exam_freq>0` 但 source 无真题题号 → 警告  

错误在入库前死掉，而不是考前才发现笔记不可信。

## 6. 下个迭代只做三件事（考生版优先级）

1. **填满机电实务至少一章真内容**（含真题 `exam_freq`），证明闭环能提分，而不是只证明脚本能跑。  
2. **`make today` + `make anki-sync` 一键化**，工作日 1 分钟内开工。  
3. **客观题模板 + 真题卷面案例骨架**，对齐一建真实得分方式。  

做完这三件，才值得把「主笔记和错题本」迁进本系统；否则仍会：教材 PDF 划线 + 商业题库刷题 + 自建 Anki——系统再优雅也只是旁路。

---

## 7. 总结

aistudy 不是「AI 生成一堆笔记」的玩具。它抓住了一建备考最难的三件事：**可追溯、可诊断、可复练**。  
机电实务的「参数 / 规范条文 / 工序 / 案例采分」被数据模型与 skill 护栏钉死，这是正确的考试工程学。

但考生不给「架构分」，只给「能不能涨分」。  
**当前最大缺口不是再多一个 skill，而是：可考的内容密度、更短的每日路径、以及与真题/题型对齐。**  

把仓库从「电缆敷设 demo」变成「机电实务可作战知识库」之日，才是本系统对一建考生真正有用之时。
